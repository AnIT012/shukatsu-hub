# 就活Hub

毎朝開いて「**次に何をすべきか・次の締切はいつか**」が一目でわかる、就活の進捗管理 PWA。
選考フローを **段階 ＞ タスク** で整理し、説明会・イベントもまとめて管理。努力の積み上げを褒める「進捗」ページ付き。スマホでアプリのように使える（ホーム追加・オフライン対応・プッシュ通知）。

> 本選考もインターンも、ESもWebテストも面接も。「いま動くべきこと」だけが前に出る設計。情報量を絞り、迷わせない。

---

## 主な機能

### 選考フロー（段階 ＞ タスク）
- **段階**（書類選考・一次面接 …）の中に **タスク**（ES提出・Webテスト・面接 …）を持つ2層モデル
- タスクの 〇 はタップで進める：**未 → 提出済 → 完了**
  - 締切と実施日が**両方ある**タスクは「提出済（締切を消化＝注目が実施日へ移る）→ 完了」の2段階
  - 片方だけなら「未 → 完了」の一発
- 段階ごとに **通過 / 不合格 / 辞退** を選ぶ（全体結果は段階から自動導出）
- **並行対応**：ES＋Webテストなど同時に進む選考を1つの段階にまとめられる
- **見る / 編集モードの分離**：ふだんは見るだけ（〇と結果）でスッキリ。「編集」で段階の追加・並べ替え・締切/メモ編集
- 「結果待ち」はどの段階で待っているか（例：一次面接 の結果待ち）まで表示

### 一覧（選考・イベント）
- **コンパクト / 詳細** の表示切替（端末ごとに記憶）。コンパクトは「締切・企業・次にやること＋ピンリンク」だけ、詳細は進捗バー（段階の色）も
- 結果の見せ方：内定/参加確定＝緑枠、不合格・辞退＝控えめ、結果待ち＝時計
- **直近1週間の予定**を常設の固定枠で表示（今週内は赤・先はテーマ色）
- 締切順 / 優先度順 / 企業名順、昇降、状況・優先度での絞り込み

### 説明会・イベント
- 申込締切・開催日・形式・複数リンク・メモを管理
- 未参加 / 参加済 / 辞退 をワンタップ。開催日が過ぎたら自動で完了扱い

### 進捗（積み上げ）ページ
- 合格数ではなく **「ESを何枚出した・説明会に何回参加・Webテスト・面接GD」=努力そのもの** を可視化
- 努力（歩数）で **花畑が育つ**（土 → 芽 → … → 満開）。**称号バッジ**も解放
- メイン上部の伴走コメントから1タップで開ける「ご褒美」ページ

### 体験
- 下タブ **[進捗][選考][イベント][設定]** ＋左右スワイプ切替
  - アイコンが選択で**進化＋動く**：選考＝チェックがポンと入る / イベント＝ベルが揺れる / 進捗＝芽→花
- メイン上部に**伴走コメント**（選考・イベントで別文言・テーマ追従・日替わり）
- テーマ12色（和カラー）＋フォント選択、ライト/ダーク対応
- 操作に合わせた控えめなマイクロアニメーション

### データと同期・安全性
- **クラウド同期**（Supabase Auth ＋ Postgres、1ユーザー1行の JSON）
- **多端末の上書き事故を防止**：書き込みは**楽観ロック（自分が読んだ版と一致する時だけ更新）**。別端末が更新していたら上書きせず取り込み、未送信分は復元ポイントへ退避
- **自動ローカルバックアップ**：保存のたびに直近世代を端末に退避。設定 > データ > 復元ポイントからいつでも戻せる
- **オフライン対応**：オフラインでも編集→端末に保存→オンライン復帰で自動同期
- **Web Push 通知**：締切・予定のリマインド（毎日まとめ / 指定日数前）。各ユーザーの設定時刻に配信
- JSON エクスポート / インポートでバックアップ・端末移行

## セットアップ

```bash
npm install
cp .env.local.example .env.local   # Supabase を使う場合は値を設定
npm run dev                         # http://localhost:3000
```

- **ローカルのみ**で使う場合：環境変数なしでも起動でき、データはブラウザの `localStorage` に保存される
- **クラウド同期 / 通知**を使う場合：Supabase の設定が必要（→ [`SETUP-CLOUD.md`](./SETUP-CLOUD.md)）
  - `supabase/feedback.sql` … フィードバック用テーブル（SQL エディタで1回実行）
  - `supabase/functions/notify` … 通知配信の Edge Function（毎時 pg_cron で起動し、各ユーザーの設定時刻に送信）

```bash
npm run build   # 本番ビルド
npm start        # 本番起動
```

## アーキテクチャ（要点）

データ型は [`src/lib/types.ts`](./src/lib/types.ts) を参照。

```
Application（1社）
 ├─ stages: SelectionStage[]      段階(>タスク)。中核モデル
 │   └─ tasks: SelectionTask[]    並行なら複数（submitted=提出済 / done=完了）
 ├─ steps:  SelectionStep[]       旧モデル(移行ソースとして保持＝バックアップ)
 ├─ esEntries / links / memo …
EventItem（説明会・イベント）
```

- **次のアクション / 進捗 / 状況分類 / 努力集計**：[`src/lib/next-action.ts`](./src/lib/next-action.ts)
  - 注目日は `focusOf`（締切優先・提出済/超過で実施日へ）。`effortSummary` で努力ページの集計
- **データ正規化と移行**：[`src/lib/io.ts`](./src/lib/io.ts)
  - 読み込み時に欠損フィールドを補完（後方互換）。旧 `steps[]` から `stages[]` を自動生成（**旧データは消さず保持**）。全体結果は段階から導出。旧イベントの単一 URL は複数リンクへ移行
- **保存 / 同期 / オフライン / 楽観ロック**：[`src/lib/store.tsx`](./src/lib/store.tsx)（端末キャッシュを真実として dirty 管理、`updated_at` の条件付き更新で衝突回避、復帰時に flush / pull）
- **自動ローカルバックアップ**：[`src/lib/snapshots.ts`](./src/lib/snapshots.ts)
- **移行通告**：[`src/components/version-notice.tsx`](./src/components/version-notice.tsx)（旧データ検出で1回だけ案内＋スナップショット退避）

### 安全性
- ユーザー入力 URL は `http(s)` のみ許可（`safeHref` で `javascript:` 等を無効化）
- Supabase は RLS で user_id ごとにデータを分離。クライアントには VAPID 公開鍵のみ（秘密鍵は Edge Function の secrets）
- 破壊的操作（全削除・復元）は確認ダイアログ必須。書き込みは楽観ロック＋復元ポイントで取り返しがつく
